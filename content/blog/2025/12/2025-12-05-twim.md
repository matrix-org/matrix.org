### Website & Content WG Report ([website]((@/foundation/working-groups/index.md)))

It's been [a while](@/blog/2025/09/2025-09-26-twim.md) since the [last status update](@/blog/2025/10/2025-10-24-twim.md) from the [Website & Content WG](@/foundation/working-groups/index.md) if we disregard the [recent upgrade to the TWIM room and bot](@/blog/2025/11/2025-11-21-twim.md), so we wanted to share a quick wrap-up of recent developments!

- We continue to explore how to best and most effectively collaborate with the Foundation on publishing news, with the [FOSDEM announcement](@/blog/2025/12/2025-12-03-fosdem-presence.md) by the Events WG following the newest suggested process with success!
- We have suggested to adopt an [AI policy](https://github.com/matrix-org/matrix.org/blob/main/CONTRIBUTING.md#ai-policy) to the Foundation, and it was decided to adopt it on the website for now.
- In collaboration with the [Room Directory WG](@/foundation/working-groups/index.md), you can now find the form as well as further information about the room directory of the matrix.org homeserver [on a dedicated page](@/homeserver/room_directory.md)
- weeman has added TWIM to the website navbar, so you don't have to remember the shortlink anymore to get your news!
- vedant-vijay fixed an issue with the filters on the [SDKs page](@/ecosystem/sdks/index.md)
- Jack S. has spotted some bugs and opportunities to polish which we were glad to merge!
- We are maintaining the information about the Foundation, including it's [members](@/membership/_index.md) and are working with the Governing Board to keep its documentation up to date, as well as the Working Groups
- We have introduced a bunch of labels to the website issue tracker to help us sort tasks, and especially to [point out](https://github.com/matrix-org/matrix.org/issues?q=is%3Aissue%20state%3Aopen%20label%3A%22help%20wanted%22) some tasks that would be easy to implement even by external contributors
- We keep developing guidelines for ourselves to improve the consistency and professionality of the website [step by step](https://github.com/matrix-org/matrix.org/issues/3030) and of publications in general step by step, while also trying to keep an eye on making it as [simple as possible](https://github.com/matrix-org/matrix.org/issues/3034) to start contributing
- The [#matrix.org-website:matrix.org](https://matrix.to/#/%23matrix.org-website%3Amatrix.org) room was upgraded and as part of that not only receives GitHub webhooks about activities on the matrix.org repo, but also the other repos required for the WG's daily work, including twim-config and the conference website repo
- Of course, we continue to publish TWIM every Friday üòâ

As you can see, there is a lot going on! If you're interested in getting involved, you can find us in the [#matrix.org-website:matrix.org](https://matrix.to/#/%23matrix.org-website%3Amatrix.org) room, or of course [grab an issue](https://github.com/matrix-org/matrix.org/issues?q=is%3Aissue%20state%3Aopen%20label%3A%22help%20wanted%22) and send a PR! We also try to meet in a call regularly on Thursday afternoons, which we announce at [#matrix.org-website:matrix.org](https://matrix.to/#/%23matrix.org-website%3Amatrix.org).

### Tuwunel ([website](https://github.com/matrix-construct/tuwunel))

Enterprise successor to conduwuit, the high-performance and feature-rich fork of Conduit.

[jason üéî](https://matrix.to/#/@jason:tuwunel.me) reports

> Tuwunel is now deployed at scale serving the citizens of Switzerland in production. Some optimizations were requested to reduce operating costs from projected growth over product lifespan: this release delivers with markedly reduced CPU usage and improves responsiveness.
> 
> #### ‚ú® New Features For Version 1.4.7
> 
> - Upgrade Support for Room Version 12 is now available. Though this room version has been supported for the entire 1.4 series, all Tuwunel servers have been protected by Hydra Backports on all room versions. As such, other work was able to be prioritized for the preceding releases.
> 
> #### üìà Enhancements
> 
> - Recursive relations have been enabled. This is an optimization which allows the server to gather more data using fewer client requests, for example, of a thread with many reactions and replies. The implementation is now optimal and utilizes the full capabilities of Tuwunel's asynchronous database.
> - Several miscellaneous but significant optimizations took place at the direction of memory profiling. This reduced load on the allocator for database queries and JSON serialization of complex objects. Heroes calculations and the joined room hot-path on sync v3 were further optimized to reduce the database query load itself.
> - Jemalloc has been repackaged with platform-specific optimizations enhancing the build. The upgrade to the dev branch of libjemalloc itself was considered as too much variability for the same release, it is planned for an upcoming release.
> - Thanks to element-hq/synapse#18970 by dasha-uwu, we have very slightly turned down the amount of randomness when selecting join-servers, More retries also occur within a single request if necessary. Thanks to gogo199432 and lifeofguenter for reporting problems in (#128) and (#205) respectively. More opportunities are still available to make large room joins robust.
> 
> #### üêû Bug Fixes
> 
> - Special thanks to yefimg for fixing LDAP logout in (#231) from a report kindly made by orhtej2 in (#97); thank you for your patience waiting for domain expertise to assist here.
> - Thanks to Radiant-Xyz the example configurations have been updated to remove allow\_check\_for\_updates. This fixes any warnings for the item no longer existing. (#221)
> - Thanks again to Radiant-Xyz reporting in (#219) the /whoami endpoint now returns spec-compliant errors for Mautrix bridges (fe12daead9). Thanks also to @bobobo1618 for confirming the fix is working.
> - Relations responses were sometimes incorrect in the forwards direction. This was fixed by (5147b541) bringing those responses into full compliance. Note the prior release had also fixed compliance issues but in the backwards direction.
> - Server selection for backfill struggled sometimes for version 12 rooms. These rooms might fail to load history after join. Additional servers are now found using creators and additional\_creators instead.
> - Room leave compliance has been fixed for an edge-case where a room becomes empty except for a locally invited user which does not have its leave event sent down /sync.
> - Thanks to grinapo for a report which lead to the discovery of events acquired over backfill not being checked for whether they already exist.
> 
> #### ‚è© Upcoming
> 
> Several planned items could not be cut into this release: SSO/OIDC support (#7), Element Call setup assistance and documentation (#217)(#215), User-level Admin Room and Media deletion (#192), and any other assigned issue. These items are on the short-list for the next cycle and mean a lot to us. To all participants: your issues are not being ignored and we hear you.
> 
> #### üíú Tuwunel Needs Your Support üíú
> 
> We are looking for additional sponsorship so June can work on Tuwunel again full-time. If you are a company looking to accelerate development with features you need please email <june@girlboss.ceo> right away.
>

### Element X Android ([website](https://github.com/vector-im/element-x-android))

Android Matrix messenger application using the Matrix Rust SDK and Jetpack Compose.

[Jorge](https://matrix.to/#/@jorgem:element.io) says

> In the Element X Android team, this week we've been focused on:
> 
> - Spaces! I know, I know, every week we work on spaces, but it's a huge task because we really want to have the best UX possible.
> - Improve the UX for opening a room: instead of loading the joined room data while we're navigating, which can result in some flickering, we now preload the room and then navigate with the result.
> - Added room list indicators about messages that are being sent or failed to send. There is also some ongoing work for displaying edits and reactions properly.
> - Enabled the Sentry Rust SDK inside our SDK so we can get info about panics and other issues coming directly from the Matrix Rust SDK.
> - Worked on doing some periodic cleanup step in the SDK databases to ensure they remain performant.
> - Fixed some issues related to voice recoding, ignored users, the rich text editor sizing and the text inside notifications for spaces.

### matrix-rust-sdk ([website](https://github.com/matrix-org/matrix-rust-sdk))

Your all-in-one toolkit for creating Matrix clients with Rust, from simple bots to full-featured apps, with bindings to Swift, Kotlin, WebAssembly, Go and more, sponsored by Element.

[Ivan ü¶Ä](https://matrix.to/#/@mnt_io:matrix.org) says

> Hello everyone! This week, we've released the version¬†0.16.0 of the SDK!
> 
> You may be wondering what happened to the 0.15.0¬†release, and you‚Äôre right to ask. Unfortunately, it had to be yanked after a problematic Git dependency slipped through during publishing.
> 
> This created a situation where some SDK crates were successfully released as 0.15.0, while others failed to publish. To avoid confusion and maintain consistency, the entire 0.15.0 release was yanked.
> 
> See more on [the release page](https://github.com/matrix-org/matrix-rust-sdk/releases/tag/matrix-sdk-0.16.0).
> 
> #### Latest Event
> 
> - [#5913](https://github.com/matrix-org/matrix-rust-sdk/pull/5913) Edits are now supported as part of the Latest Event! Finally, one might say, and to which we can only agree.
> - [#5624](https://github.com/matrix-org/matrix-rust-sdk/pull/5624) The old Latest Event API has been removed. -3001¬†lines of code, yepee.
> 
> #### Timeline
> 
> - [#5900](https://github.com/matrix-org/matrix-rust-sdk/pull/5900) The Timeline can now be configured to hide read receipts on state events. Not all clients expect this behaviour, so this is a configuration, see the new `TimelineReadReceiptTracking` type.
> 
> #### Send Queue
> 
> - [#5920](https://github.com/matrix-org/matrix-rust-sdk/pull/5920) The Send Queue now inserts newly sent events in the Event Cache directly! It avoids the situation where a sent event could disappear: it is no longer a local event but not yet a remote event if the server lags to give it back via the `/sync`. Now we no longer wait on the server, the event is already here, period.
> 
> #### Client
> 
> - [#5916](https://github.com/matrix-org/matrix-rust-sdk/pull/5916) Support refreshing the access token when fetching the server versions. A deadlock has also been removed. One stone, two problems.
> - [#5911](https://github.com/matrix-org/matrix-rust-sdk/pull/5911) We are improving performance of the stores relying on SQLite. We've noticed that fragmentation is a non-negligible performance bottleneck. This patch introduces `Client::optimize_stores` to optimise‚Ä¶ well‚Ä¶ the stores. This is an exploration to see how to defragment the database without disturbing your apps too much. The impact is particularly noticeable on Android devices.
> - [#5917](https://github.com/matrix-org/matrix-rust-sdk/pull/5917) Support for the stable `m.oauth` UIAA type has been added, as it has recently landed in [the Matrix specification](https://github.com/matrix-org/matrix-spec/pull/2234).
> 
> #### Room List
> 
> - [#5927](https://github.com/matrix-org/matrix-rust-sdk/pull/5927) The Room List now receives less updates, which reduces the number of redraws app-side.
> 
> #### Bindings
> 
> - [#5919](https://github.com/matrix-org/matrix-rust-sdk/pull/5919) A network request was sent only for log purposes. It was breaking the offline mode detection, and was slowing down the startup on slow network conditions. This request has been removed as it was considered useless!
