> Here's your weekly spec update! The heart of Matrix is the specification - and this is modified by Matrix Spec Change (MSC) proposals. Learn more about how the process works at <https://spec.matrix.org/proposals>.
>
> ### MSC Status
>
> **New MSCs:**
>
> - [MSC4497: Filter state events by type in /state endpoint](https://github.com/matrix-org/matrix-spec-proposals/pull/4497)
> - [MSC4496: Calendar Events, Invites, and Availability in Matrix](https://github.com/matrix-org/matrix-spec-proposals/pull/4496)
>
> **MSCs in Final Comment Period:**
>
> - [MSC4127: Removal of query string auth](https://github.com/matrix-org/matrix-spec-proposals/pull/4127) (merge)
>
> **Accepted MSCs:**
>
> - [MSC4186: Simplified Sliding Sync](https://github.com/matrix-org/matrix-spec-proposals/pull/4186)
>
> **Closed MSCs:**
>
> - [MSC4418: Make `destination` a required server authentication field](https://github.com/matrix-org/matrix-spec-proposals/pull/4418)
>
> ### Spec Updates
>
> Hot off the heels of [MSC2545: Image Packs](https://github.com/matrix-org/matrix-spec-proposals/pull/2545) being accepted last week (and [merged to the spec](https://github.com/matrix-org/matrix-spec/pull/2397) this week!), [MSC4186: Simplified Sliding Sync](https://github.com/matrix-org/matrix-spec-proposals/pull/4186) was accepted by the SCT on Tuesday!
>
> Note that this is only the core sliding sync MSC: all the extension MSCs are still in review. Sliding Sync won't land in the spec until at least a couple of those are accepted as well (several support E2EE messaging, for instance).
>
> Regardless, SSS is a large pillar of the ongoing [Matrix 2.0](https://matrix.org/blog/2024/10/29/matrix-2.0-is-here/) work, and something that homeservers and clients already implemented and rely on (Element X), so it's really great to see the huge core MSC land!
>
> Finally, keeping (roughly) with our quarterly release cadence, expect a new Matrix spec release (v1.19) to land very shortly new with lots of exciting changes!

## Dept of Servers 🏢

### Zendrite

[bones_was_here](https://matrix.to/#/@bones_was_here:xonotic.org) says

> Zendrite has made nice progress since the [last TWIM post](/blog/2026/03/2026-03-13-twim), here's a summary.
>
> [3.1.0 has been released](https://codefloe.com/pat-s/zendrite/releases) and includes a **security update** for vulnerabilities in the image library used for media thumbnails, for the CVE list please [refer to the PR](https://codefloe.com/pat-s/zendrite/pulls/239).
>
> #### Breaking changes in 3.0.0
>
> - auto-purge empty rooms [#160](https://codefloe.com/pat-s/zendrite/issues/160) [#162](https://codefloe.com/pat-s/zendrite/issues/162)
>
> These features automatically delete old rooms, by default it's done when all local users have forgotten a room, you may also wish to enable [auto-forgetting rooms on leave](https://zendrite.pat-s.me/administration/auto-forget-on-leave/) and [auto-purging empty rooms](https://zendrite.pat-s.me/administration/auto-purge-empty-rooms/).
>
> #### Other features
>
> - **nats:** Add NATS authentication and credential rotation support [#228](https://codefloe.com/pat-s/zendrite/issues/228)
> - **clientapi:** implemented registration tokens [#175](https://codefloe.com/pat-s/zendrite/issues/175)
> - add MatrixRTC transports endpoint (MSC4143) [#122](https://codefloe.com/pat-s/zendrite/issues/122)
>
> #### Optimisations
>
> - push required_state filters into SQL to avoid loading all room state [#121](https://codefloe.com/pat-s/zendrite/issues/121)
> - fetch large room state in chunks to prevent OOM during state resolution [2a1a3a8](https://codefloe.com/pat-s/zendrite/commit/2a1a3a8)
>
> #### Bigger fixes
>
> - panic on malformed federated events (affects Dendrite too, could cause "boot-looping") [#216](https://codefloe.com/pat-s/zendrite/issues/216)
> - when purging a room try to leave cleanly first, and invalidate cache (affects Dendrite too) [#158](https://codefloe.com/pat-s/zendrite/issues/158)
> - allow outgoing federation to use ipv6 by default (affects Dendrite too) [#157](https://codefloe.com/pat-s/zendrite/issues/157)
> - roomserver: apply redactions from privileged creators in v12+ rooms (affects Dendrite too) [#151](https://codefloe.com/pat-s/zendrite/issues/151)
> - syncapi: fix /messages sender/type filtering (affects Dendrite too), **important for moderation bots** [#130](https://codefloe.com/pat-s/zendrite/issues/130)
>

### Tuwunel ([website](https://github.com/matrix-construct/tuwunel))

Enterprise successor to conduwuit, the high-performance and feature-rich fork of Conduit.

[june ✿ 🏳️‍⚧️ (it/she/puppy)](https://matrix.to/#/@june:woof.gay) says

> [Tuwunel](https://github.com/matrix-construct/tuwunel) 1.8.0 opens an existing Conduit database directory and migrates it in place. June's [migration](https://github.com/matrix-construct/tuwunel/issues/41) carries over rooms, media including S3-backed objects, pending knocks, and suspended or locked users, and attributes media to its local owners. It is the largest single capability this cycle, raised by Korvox.
>
> Identity and authentication grew alongside it. [MSC2290](https://github.com/matrix-org/matrix-spec-proposals/pull/2290) third-party-identifier email support arrives with outbound mail, storage, and management endpoints. Matrix Authentication Service support lands too: the provisioning API, synced email bindings, and [MSC3824](https://github.com/matrix-org/matrix-spec-proposals/pull/3824) SSO redirect-action forwarding, opened by CEbbinghaus. dasha-uwu added a [remote-server version API](https://github.com/matrix-construct/tuwunel/commit/d038c5bc8) and `SSLKEYLOGFILE` support for debugging federation TLS.
>
> Spec compliance advanced across [MSC2666](https://github.com/matrix-org/matrix-spec-proposals/pull/2666) mutual_rooms, [MSC3765](https://github.com/matrix-org/matrix-spec-proposals/pull/3765) rich-text topics, [MSC4133](https://github.com/matrix-org/matrix-spec-proposals/pull/4133) extended profiles, and [MSC4311](https://github.com/matrix-org/matrix-spec-proposals/pull/4311) invite and knock state as full PDUs, plus an [MSC1692](https://github.com/matrix-org/matrix-spec-proposals/pull/1692) registration terms stage opened by erebion and [MSC4190](https://github.com/matrix-org/matrix-spec-proposals/pull/4190) appservice device management opened by ngophuocloi-miracle-aavn, all tracked in the [compliance list](https://github.com/matrix-construct/tuwunel/blob/55d59989c0af57860a2afdfabf4c210ef66e499c/docs/development/compliance/msc.md).
>
> One fix comes with an apology. A 1.7.x allow-list that omitted the local server name [dropped local users' own events](https://github.com/matrix-construct/tuwunel/issues/489) as forbidden remote traffic; the local server is now always exempt. Reported by BurningEnlightenment. timestamp_to_event ([MSC3030](https://github.com/matrix-org/matrix-spec-proposals/pull/3030)) no longer returns M_NOT_FOUND on valid searches, [fixed](https://github.com/matrix-construct/tuwunel/pull/477) by lingbohome, and falls back to federation for local misses. Lazy-loaded incremental sync keeps changed members ([MSC4222](https://github.com/matrix-org/matrix-spec-proposals/pull/4222)).
>
> Operators upgrading should note: a push gateway on a private or loopback address now needs an `ip_range_denylist` exception, the RocksDB engine moves to 11.1.1, and the first 1.8.0 boot rebuilds the timestamp index once, slowing first startup.
>
> Full release notes: <https://github.com/matrix-construct/tuwunel/blob/55d59989c0af57860a2afdfabf4c210ef66e499c/RELEASE.md>

### Element X iOS ([website](https://github.com/vector-im/element-x-ios))

A total rewrite of Element iOS using the Matrix Rust SDK underneath and targeting devices running iOS 17+.

[Mauro](https://matrix.to/#/@mauro.romito:element.io) reports

> * It’s been a feature requested for such a long time, and finally, user status work has been started.
> * Various accessibility issues were fixed, and more are to come.
> * We moved search into its own dedicated search tab to reduce cluttering and give a cleaner UX.
> * And to stay in the theme of search… we’re also currently working on a feature that allows users to search among all their messages, another long-requested feature that is being worked on.
> * The app has been migrated to the new Swift 6.2 version of the language, ensuring that the concurrent code of our app is validated at compile time. This has improved the performance and stability of the app by a lot.
> * The ability to create polls that allow for the selection of multiple answers has been added.
