### matrix-rust-sdk ([website](https://github.com/matrix-org/matrix-rust-sdk))

Your all-in-one toolkit for creating Matrix clients with Rust, from simple bots to full-featured apps, with bindings to Swift, Kotlin, WebAssembly, Go and more, sponsored by Element.

[Ivan ðŸ¦€](https://matrix.to/#/@mnt_io:matrix.org) reports

> Happy New (Gregorian) Year folks! Back at work after a couple of weeks off for the team and most of the contributors. Snow is falling outside, cold is here to last, kids are building igloos, let's see what happens in the Matrix Rust SDK recently. 
> 
> #### Latest Event
> 
> We are polishing the last bits, but the meta issue can be closed. The API is considered stable and fulfill its promises.
> 
> - [#6035](https://github.com/matrix-org/matrix-rust-sdk/pull/6035) The `LatestEventValue` is erased when a room has been emptied. This is useful when a user is ignored: the `LatestEventValue` will automatically be re-computed.
> - [#6016](https://github.com/matrix-org/matrix-rust-sdk/pull/6016) Some errors from the Send Queue are recoverable, and must not mark the `LatestEventValue` as `LocalCannotBeSent`, but must keep being a `LocalIsSending`.
> 
> #### Sliding Sync
> 
> We have cleaned up the Sliding Sync API a bit to remove the sticky parameters feature entirely, making `request.room_subscriptions` non-sticky every time. Also, once a room subscription is sent, it's not necessary to send it again, thus reducing the size of the request payloads.
> 
> - [#6002](https://github.com/matrix-org/matrix-rust-sdk/pull/6002) and [#6009](https://github.com/matrix-org/matrix-rust-sdk/pull/6009) The `sliding_sync::sticky_parameters` has been entirely removed (an artifact from MSC3575).
> - [#6012](https://github.com/matrix-org/matrix-rust-sdk/pull/6012) New methods on `SlidingSync`: `unsubscribe_to_rooms` to unsubscribe to a set of rooms, and `clear_and_subscribe_to_rooms` to reset the subscriptions and subscription to a fresh set of rooms.
> - [#6005](https://github.com/matrix-org/matrix-rust-sdk/pull/6005) Introduce the `PollTimeout` type to control the `timeout` query parameter to `/sync` with more delicacy.
> 
> #### Space
> 
> A lot of work is happening on the Space API, but a single PR has been merged:
> 
> - [#6010](https://github.com/matrix-org/matrix-rust-sdk/pull/6010) Add `CreateRoomParameters::is_space` in the bindings.
> 
> #### SQLite
> 
> Our quest to improve SQLite performance is pursuing:
> 
> - [#6004](https://github.com/matrix-org/matrix-rust-sdk/pull/6004) Add WAL checkpoints when creating DB pools and vacuuming. To quote the author:
> 
>   > We've noticed the WAL files growing larger and larger, not shrinking as expected. This may be due to connections being busy and not allowing the automatic WAL checkpoints to perform this shrinking operation or some other causes, we're still investigating the issue.
>
>   >
>   > In the meantime we can manually add WAL checkpoints:
>   >
>   > 1. When creating the DB pools, before any read/write operations are done.
>   > 2. Before vacuuming, so the WAL has room to grow to match the DB.
>   > 3. After vacuuming, since the copied data in the WAL file has been stored in the DB.
> 
> #### RTC
> 
> - [#6003](https://github.com/matrix-org/matrix-rust-sdk/pull/6003) Add new call intents for voice only.
> 
> #### Regular dose of bug fixes
> 
> And the most notable bug fixes of the weeeeek aaaaaaaaare:
> 
> - [#6027](https://github.com/matrix-org/matrix-rust-sdk/pull/6027) Fix a double `msgtype` in a `m.location` event (a bug from `ruma`).
> - [#5996](https://github.com/matrix-org/matrix-rust-sdk/pull/5996) Use the server name from the user ID as a fallback URL for fetching the well-known info.
> - [#5994](https://github.com/matrix-org/matrix-rust-sdk/pull/5994) Deduplicate aggregation local and remote echo in the Timeline.
> - [#5992](https://github.com/matrix-org/matrix-rust-sdk/pull/5992) Create key dirs if they don't exist in the search engine.
> - [#5858](https://github.com/matrix-org/matrix-rust-sdk/pull/5858) Utilize the Event Cache and include common relations when focusing on an event without context.
