### pygomx ([website](https://codeberg.org/saces/pygomx))

A golang matrix high level client library with c-bindings for python

[saces](https://matrix.to/#/@saces:c-base.org) announces

> I want a library that hides all the matrix 2.0 from me and let me build my business logic in python and do quick&dirty (matrix)helper scripts. I did not found any I wanted to use, so I built my own, but with ‚Äî no I'm not Benderü§ñ ‚Äî a simple text based c-api and multi client support:
> 
> - project home: <https://codeberg.org/saces/pygomx>
> - matrix room: [#pygomx:matrix.org](https://matrix.to/#/#pygomx:matrix.org)
> 
> pygomx is a monorepos to keep the right versions together and consists of three parts:
> 
> ##### libmxclient - golang matrix client library
>
> - High level client library with a simple c-api: it basically passes (json)strings back and forth.
> - All the Matrix 2.0 magic is hidden in the lib.
> - Should work on *any* platform golang is enable to build for.
> 
> ##### pygomx-module - python binding package
>
> - This package contains only the python bindings module binary
> - Should work on any platform you have a decent python and c-compiler for.
> 
> ##### smal - simple matrix application library
>
> - Matrix library and tools that utilizes the bindings
> - Inspired by [Simple-Matrix-Bot-Lib](https://codeberg.org/imbev/simplematrixbotlib)
> - Example tool: 
> 
> ```python
> import sys
> from _pygomx import lib, ffi
> import click
> import json
> 
> 
> @click.command()
> @click.argument("domain", metavar="string")
> def discoverhs(domain):
>     """Attempts to discover the homeserver from the given string"""
>     mxid = domain.encode(encoding="utf-8")
> 
>     r = lib.cli_discoverhs(mxid)
>     result = ffi.string(r).decode("utf-8")
>     lib.FreeCString(r)
>     if result.startswith("ERR:"):
>         print(result)
>         sys.exit(1)
>     result_dict = json.loads(result)
>     print(result_dict["m.homeserver"]["base_url"])
> ```

> - Minimal bot example:
> 
> ```python
> from smal.bot import SMALBot
> DEFAULT_PREFIX = "!"
> 
> class MinimalBot(SMALBot):
>     def on_sys(self, ntf):
>         print("Got a system notification: ", ntf)
>     def on_event(self, evt):
>         print("Got a matrix event: ", evt)
>     def on_message(self, msg):
>         print("Got a room massage: ", msg)
> 
> def main():
>     # create and initialize the bot
>     bot = MinimalBot(DEFAULT_PREFIX)
>     # start syncing forever (listen for incommmig messages/events)
>     bot.run()
> 
> if __name__ == "__main__":
>     main()
> ```
> 
> ---
> 
> The project is at an early stage, things are still subject to change, but the basics are done and working.  
> Let's see where it evolves.
> 
> Help would be welcome for python packaging, especially for building anywheels.
> 
> Any other kind of help will be welcome too, of course ;)
> 
> That's all for now, see you.
  
### matrix-rust-sdk ([website](https://github.com/matrix-org/matrix-rust-sdk))

Your all-in-one toolkit for creating Matrix clients with Rust, from simple bots to full-featured apps, with bindings to Swift, Kotlin, WebAssembly, Go and more, sponsored by Element.

[Ivan ü¶Ä](https://matrix.to/#/@mnt_io:matrix.org) announces

> Salut¬†! It's been a while since our last report. Time flies and we are pretty busy. I'll try to keep it short!
> 
> #### Event Cache
> 
> - [#6190](https://github.com/matrix-org/matrix-rust-sdk/pull/6190) A pretty subtle bug has been fixed where a batch token could have been ditched in rare situations (when the sync only returns events that have been inserted inside the Event Cache via the Send Queue). The impact of this bug was missing events, so, rather important.
> - [#6157](https://github.com/matrix-org/matrix-rust-sdk/pull/6157) An undecrypted event in a `ThreadSummary` could be decrypted later, but the `ThreadSummary` wasn't aware of that: showing a UTD whilst it wasn't. Now it's no longer the case!
> - [#6085](https://github.com/matrix-org/matrix-rust-sdk/pull/6085) The pinned events are now persisted in the storage.
> 
> #### Latest Event
> 
> - [#6171](https://github.com/matrix-org/matrix-rust-sdk/pull/6171) `LatestEventValue::RemoteInvite` is computed once per room, avoiding a bug where receiving some events in an invite room could have refreshed the `LatestEventValue`'s timestamp, thus moving the room to the top of the Room List.
> - [#6143](https://github.com/matrix-org/matrix-rust-sdk/pull/6143) Not all `m.room.member` events are equal. We have restricted when `m.room.member` represents a `LatestEvent` candidate.
> - [#6096](https://github.com/matrix-org/matrix-rust-sdk/pull/6096) An edit of an event was reflected in the `LatestEventValue`, but not an edit of an edit. Well, the approach to handle edit aggregation has been revisited.
> 
> #### Logs
> 
> - [#6162](https://github.com/matrix-org/matrix-rust-sdk/pull/6162) When the logs have accumulated and were too large, they could have been rejected by the rageshake servers. We've replaced `RollingFileAppended` by a custom `SizeAndDateRollingWriter`.
> - [#6140](https://github.com/matrix-org/matrix-rust-sdk/pull/6140) A new tool to help analyse the logs produced by `matrix_sdk_ffi`: `cargo xtask log overview`. It displays the logs in a tree of targets and log levels.
> 
> #### Stores
> 
> - [#6160](https://github.com/matrix-org/matrix-rust-sdk/pull/6160) Not all environments need a cross-process lock, such as Android where an app usually runs in a single process (a contrario of iOS where notifications live in a different process than the main app). To improve performances, we've created `CrossProcessLockConfig` to configure the number of process and to act accordingly.
> - [#6130](https://github.com/matrix-org/matrix-rust-sdk/pull/6130) Creating a notification client from a regular client wasn't correctly separating the data, some were wrongly shared. This has been fixed by moving `BaseClient::room_info_notable_update_sender` into `BaseStateStore`. It breaks a couple of API, but they are simpler now.
> 
> #### Sync
> 
> - [#6148](https://github.com/matrix-org/matrix-rust-sdk/pull/6148) `""`, `null`, or simply unset, this problem with JSON and the Matrix specification seems endless. Fix a deserialisation failure when the `avatar_url` is `null`.
> - [#6136](https://github.com/matrix-org/matrix-rust-sdk/pull/6136) Exclude service members from room names when using Sliding Sync to get notifications.
> - [#6126](https://github.com/matrix-org/matrix-rust-sdk/pull/6126) A user was unaware they've been invited in a room they have previously left. The SDK knew, but the update wasn't broadcasted correctly. This is now fixed!
> 
> #### FFI
> 
> - [#6122](https://github.com/matrix-org/matrix-rust-sdk/pull/6122) The `NotificationItem` now includes the raw JSON of the underlying event.
> - [#6116](https://github.com/matrix-org/matrix-rust-sdk/pull/6116) Do not override the `package_name` and the `cdylib_name` value for Kotlin bindings.
> - [#6112](https://github.com/matrix-org/matrix-rust-sdk/pull/6112) and [#6115](https://github.com/matrix-org/matrix-rust-sdk/pull/6115) UniFFI checksums have been removed.
> - [#6101](https://github.com/matrix-org/matrix-rust-sdk/pull/6101) Add a `dist` profile with optimised binary size. The `matrix-sdk-ffi` library size is now 50-70MB smaller _per architecture_, resulting in very significant file size reduction for universal APKs, which have 4¬†architectures!

[evol](https://matrix.to/#/@evol:evolved.systems) says

> If you've been following Matrix for a while, you've probably noticed people mentioning Matrix's spec proposals (MSCs) by number, e.g. MSC4297.
> All MSCs are pull requests in the [official repository](https://github.com/matrix-org/matrix-spec-proposals). Often, I wanted to check what a MSC was about, but if it wasn't directly linked, I had to go to the repository and look for the MSC in there. > 
> It's very simple: typing a number after the slash (e.g. <https://msc.re/4297>) will redirect you to the right MSC. Typing nothing (<https://msc.re>) will take you to the home page of the repository. That's it!
> 
> The code is tiny and open source, check it out: <https://github.com/evoL/msc-redirect>. Join [#msc-re:evolved.systems](https://matrix.to/#/#msc-re:evolved.systems) if you'd like to chat!
